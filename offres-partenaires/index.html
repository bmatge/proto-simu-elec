Recommandation — Modèle de données « Offres partenaires »

Document de travail destiné à la DGEC (validation du schéma) et aux partenaires privés (guide de remplissage du téléservice). Il décrit comment intégrer dès maintenant les offres commerciales (EDF, Renault, Volvo, installateurs RGE, énergéticiens…) dans les simulateurs du portail sans modifier leur code à chaque ajout de partenaire.

12

RFR/part, profil ANAH, DPE, statut, type logement, énergie, etc.

1 · Motivation

Pourquoi un téléservice et un modèle stable, plutôt que des intégrations cas-par-cas avec chaque partenaire.

Le problème

Les simulateurs actuels (voiture, pompe à chaleur, poids lourd, passer à l'électrique) calculent les coûts et les aides à partir de paramètres économiques internes (prix d'achat catalogue, aides État, prix kWh, coûts d'entretien). Les offres privées qui accompagnent souvent ces décisions d'achat (bonus reprise constructeur, remise installateur, tarif kWh préférentiel, leasing à taux bonifié) ne sont aujourd'hui pas représentées.

Au-delà du sujet « afficher des offres », l'objectif politique est plus large : donner un cadre aux partenaires privés pour qu'ils s'alignent sur le vocabulaire et les seuils utilisés par l'État (RFR/part, profils ANAH, DPE, énergie actuelle…), et donc valoriser plus simplement leurs offres dans les parcours citoyens.

La solution proposée

  1. Les partenaires déclarent leurs offres dans un téléservice Démarches Numériques (ex-démarches simplifiées).
  2. La DGEC valide les dossiers, exporte un fichier JSON consolidé shared/partner-offers.json.
  3. Les simulateurs lisent ce fichier en fetch() et affichent les offres éligibles aux côtés des aides État.

Contrainte cardinale

Le vocabulaire des champs JSON doit être directement aligné sur les variables d'entrée déjà utilisées par les simulateurs : profil_anah reprend bleu/jaune/violet/rose, dpe reprend A..G, logement.statut reprend po/pb/loc, etc. Aucune transformation JS entre la déclaration partenaire et l'usage dans le calcul. Toute mauvaise modélisation maintenant = migration douloureuse plus tard.

2 · Schéma JSON

Un objet JSON = une offre. Le téléservice produit un fichier par dossier validé (granularité 1 dossier DN = 1 offre) ; la DGEC consolide les fichiers dans shared/partner-offers.json. 

// shared/partner-offers.json — un objet par offre
{
  "id": "edf-pac-serenite-2026q3",

  "partenaire": {
    "nom": "EDF",
    "siret": "552081317",
    "logo_url": "/shared/logos/edf.svg",
    "site_url": "https://particuliers.edf.fr/…"
  },

  "produit": "pompe-a-chaleur",
  "label": "Forfait Sérénité PAC",
  "description_courte": "Installation PAC clé en main…",

  "type_offre": "remise_fixe",
  "montant": 2000,
  "unite": "EUR",
  "cumul_aides_etat": true,
  "plafond_cumul_etat": null,

  "eligibilite": {
    "geo": {
      "zones": ["FR"],
      "departements": [],
      "exclusions": ["971","972","973","974","976"]
    },
    "foyer": {
      "rfr_part_max": 24000,
      "profil_anah": ["bleu","jaune","violet"]
    },
    "logement": {
      "statut": ["po"],
      "type": ["maison","appartement"],
      "dpe": ["D","E","F","G"],
      "energie_actuelle": ["fioul","gaz","propane"]
    },
    "vehicule": null,
    "poids_lourd": null
  },

  "affichage": {
    "position": "aide-spotlight-append",
    "couleur_accent": "#0066cc",
    "cta_label": "Activer cette offre",
    "cta_url": "https://demarches-numeriques.fr/…",
    "conditions_url": "https://…"
  },

  "validite": { "du": "2026-06-01", "au": "2026-12-31" },
  "source": {
    "url_declaration": "https://demarches-numeriques.fr/dossier/12345",
    "mise_a_jour": "2026-05-15",
    "valide_par": "DGEC"
  }
}

id

Identifiant unique de l'offre, format <partenaire>-<produit>-<label-kebab>-<periode>. Servira de clé de déduplication entre dossiers DN.

partenaire

Identité du déclarant : nom commercial, SIRET (pour vérification téléservice), URL logo, URL site partenaire.

produit

Énumération fermée alignée sur les simulateurs. Détermine quels simulateurs liront l'offre.

type_offre + montant + unite

Trois champs solidaires. L'unité dérive du type : remise_fixe → EUR, tarif_energie → EUR_par_kWh, financement → pct_taux_annuel.

cumul_aides_etat + plafond_cumul_etat

Si true, l'offre s'ajoute aux aides État (MaPrimeRénov', CEE, M12, etc.). plafond_cumul_etat permet de borner le total (ex. 100 000 € sur poids lourd).

eligibilite

5 sous-blocs : geo, foyer, logement, vehicule, poids_lourd. Chaque sous-bloc inutile pour le produit est null. Voir §3 pour le vocabulaire détaillé.

affichage

Indique aux simulateurs où et comment afficher l'offre. position est une énumération fermée alignée sur les patterns DSFR existants.

validite + source

Période d'application et traçabilité du dossier (URL DN, date de validation DGEC). Permet l'expiration automatique côté simulateur.

3 · Vocabulaire aligné

Toutes les énumérations sont fermées. Chaque valeur est reprise mot-pour-mot d'un attribut existant dans un des simulateurs — pas de remapping, pas d'aliasing.

Champ JSON Valeurs autorisées Source d'alignement

Règle d'or pour faire évoluer le vocabulaire

Tout ajout de valeur d'énumération nécessite une mise à jour conjointe de (1) cette page, (2) l'ADR-016 (Modèle de données offres partenaires v1), (3) le formulaire DN, (4) les simulateurs qui consomment cette énumération. C'est la contrepartie de la stabilité.

5 · Démonstration visuelle

Deux maquettes statiques qui illustrent comment une offre partenaire s'insère dans les patterns DSFR existants des simulateurs. Pas de JS — ce sont des reproductions HTML/CSS.

Cas 1 — Aide-spotlight PAC enrichi par EDF

Le pattern .aide-spotlight du simulateur pompe à chaleur affiche « L'État finance X% de votre projet » + la liste des aides État. L'offre EDF s'ajoute en tuile accolée, avec sa propre couleur d'accent (affichage.couleur_accent) pour ne pas être confondue avec une aide État.

Rendu sur /pompe-a-chaleur/ après filtrage des offres éligibles (profil bleu, fioul, maison DPE E).

Convention de couleurs

Les aides État utilisent toujours le bleu France (#000091). Les offres partenaires utilisent leur couleur d'accent propre (affichage.couleur_accent) pour rester clairement identifiables comme commerciales. Couleurs claires recommandées (jaune ambré, bleu ciel) — éviter le vert qui peut être confondu avec un signal de validation.

6 · Guide partenaire — Démarches Numériques

Cette section s'adresse aux partenaires privés qui renseignent le téléservice Démarches Numériques. Chaque champ du formulaire correspond à un champ JSON consommé par les simulateurs.

Correspondance champ DN ↔ champ JSON

Champ du formulaire DN Champ JSON cible Aide à la saisie
Raison sociale + SIRET partenaire.nom + partenaire.siret Validés automatiquement contre le registre INSEE par le téléservice.
Logo (PNG/SVG) partenaire.logo_url Format carré recommandé, fond transparent, ≤ 100 KB. Hébergé par la DGEC après upload.
Produit concerné (liste déroulante) produit Choix unique parmi 6 valeurs (voir §3). Détermine quels simulateurs afficheront l'offre.
Nom commercial de l'offre label Court, percutant (60 caractères max).
Description (zone texte) description_courte 2-3 phrases, 280 caractères max. Affiché tel quel dans la tuile.
Type d'avantage (liste déroulante) type_offre Choix parmi 7 valeurs. Conditionne les champs montant/unité affichés ensuite.
Montant / Taux montant + unite Le formulaire DN bascule entre € / €/kWh / % / % taux selon le type d'offre.
Cumulable avec aides État ? cumul_aides_etat + plafond_cumul_etat Case à cocher + champ plafond optionnel. Par défaut true, sauf opposition explicite des règles fiscales.
Sections d'éligibilité (déployées selon le produit) eligibilite.* Le formulaire n'affiche que les sous-blocs pertinents pour le produit (ex. pas de vehicule pour une offre PAC).
Validité (du / au) validite Au-delà de la date « au », l'offre disparaît automatiquement des simulateurs.
URL Conditions générales affichage.conditions_url Lien public vers le détail légal. Vérifié à l'instruction du dossier.

7 · Cycle de vie d'une offre

De la déclaration à l'expiration. Validation manuelle DGEC en v1 ; automatisation possible en v2.

ÉTAPE 01

Déclaration partenaire

Le partenaire remplit un dossier sur Démarches Numériques (un dossier = une offre). SIRET validé contre l'INSEE en temps réel.

ÉTAPE 02

Instruction DGEC

Un instructeur vérifie le SIRET, le label, les conditions, l'absence de pratique commerciale trompeuse. Validation ou demande de complément.

ÉTAPE 03

Export JSON

Les dossiers validés sont exportés en shared/partner-offers.json. Pipeline manuel en v1, automatisé en v2 (webhook DN).

ÉTAPE 04

Exposition simulateurs

Les simulateurs lisent le JSON au chargement, filtrent par produit et eligibilite, affichent selon affichage.position.

ÉTAPE 05

Expiration auto

Au-delà de validite.au, l'offre est ignorée. Aucune intervention manuelle nécessaire. Le partenaire peut redéclarer pour reconduction.

Gouvernance en v1

Validation 100 % humaine côté DGEC. Pas de score automatique, pas de mise en ligne directe. C'est la contrepartie d'avoir un cadre stable et de pouvoir refuser une offre qui ne respecte pas les principes du Plan d'électrification (par ex. promotion thermique).

8 · Limites v1 & décisions à acter

Le périmètre est volontairement restreint pour la v1. Voici ce qui est hors-scope et fera l'objet d'itérations ultérieures.

  • Granularité géographique fine — les offres sont nationales ou par département. Pas de granularité IRIS, EPCI, commune en v1. Suffisant pour les partenaires industriels nationaux.
  • Segments véhicule — éligibilité par carburant (diesel/essence), pas par modèle précis (sauf via le champ libre modeles_eligibles). Pas de référentiel constructeur intégré.
  • Barèmes dégressifs — une offre = un montant unique. Pas de barème par tranche (« −1 500 € jusqu'à 30 000 € de RFR, puis −800 € au-delà »). Pour ça : déclarer plusieurs offres avec critères rfr_part_max distincts.
  • Cumul plafonné dynamique — plafond_cumul_etat est un nombre fixe, pas une formule. OK pour M12 (100 k€) ; pas suffisant si demain une aide État a un plafond fonction du prix d'achat.
  • Stock / quotas — pas de notion de « les 1 000 premiers ». Les offres limitées en quantité doivent gérer leur retrait via validite.au.
  • Pas de personnalisation A/B — l'offre s'affiche identique pour tous les utilisateurs éligibles.
  • Ingestion non-implémentée — la consommation effective de partner-offers.json par les simulateurs sera une itération séparée, après validation du schéma.

ADR-016 à formaliser

La présente recommandation sera figée dans une ADR (« Modèle de données offres partenaires v1 ») dans le vault Obsidian, projet simulateurs-portail-elec. Cette ADR engage : tout nouveau simulateur expose ses énumérations alignées sur le schéma ; tout ajout d'énumération exige une mise à jour conjointe page + ADR + schéma DN.
